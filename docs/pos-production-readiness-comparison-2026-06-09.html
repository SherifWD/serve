POS Production Readiness and Global Benchmark Report

Project: Janova Suite POS / Restaurant POS application

Date: June 9, 2026

Workspace: /Applications/XAMPP/xamppfiles/htdocs/pos_restaurant

Scope: Backend POS API, mobile waiter/cashier/kitchen/customer flows, owner reporting endpoints, receipt PDF generation, server print queue, inventory checks, payments, tenant authorization, and scenario coverage for cafes, takeaway shops, drinks-only shops, and restaurants.

Executive verdict: The system is functionally strong for a controlled pilot or staging production trial, but it is not yet ready for wide unattended production rollout across mixed restaurant types without closing the gaps listed in this report.

The core lifecycle passed automated real-life scenario testing: order creation, KDS send, kitchen ready status, send to cashier, cash/card payment, paid-scope receipt PDF generation, print job queue, print claim, printed status, stock consumption, owner dashboard support, customer takeaway checkout, table move, and split tender.

The biggest readiness risks are cashier-only takeaway table handling, physical hardware confirmation beyond API print-job status, offline/local-first guarantees compared with Toast/Square/Oracle/NCR, and missing or partial advanced modules such as reservations, labor scheduling, native delivery marketplaces, gift cards, deep loyalty campaigns, and full load testing.

4. Real-Life Scenario Test Results

All seven scenarios below were executed through the automated feature test tests/Feature/PosProductionScenarioTest.php. Each scenario completed a paid order and a receipt print job. The print result means API-level print queue completion, not physical paper verification.

Scenario 1: Small Cafe Shop - Dine-in Table Service

Actor cycleCustomer sits -> waiter opens table -> waiter adds item/note -> waiter sends order to KDS -> kitchen marks item ready -> waiter sends order to cashier -> cashier takes cash -> cashier generates paid receipt PDF -> cashier queues receipt print -> printer device claims job -> job marked printed -> owner dashboard can include paid order.
Expected real shop behaviorNormal small cafe table-service flow where one waiter, one barista/kitchen user, and one cashier are involved.
What happenedOrder became paid. Payment status became paid. Paid receipt total was greater than zero when using scope=paid. Print job moved queued -> printing -> printed. Dine-in table status returned to open after payment.
ResultPass for controlled production flow.
Readiness noteNeeds UI/device confirmation on the actual waiter, kitchen, and cashier apps.

Scenario 2: Big Cafe Shop - Dine-in With Split Tender

Actor cycleCustomer order was entered by waiter, sent to KDS, marked ready, sent to cashier, paid with split cash/card tender, then receipt PDF and print job were completed.
Expected real shop behaviorHigher-volume cafe may have more checks and mixed payment methods. Split tender must not leave order partially paid.
What happenedSplit tender completed. Order status paid. Payment status paid. Receipt printed through queue. Table returned to open.
ResultPass for split tender API flow.
Readiness noteThis did not simulate true rush-hour concurrency, printer contention, or multiple simultaneous cashiers.

Scenario 3: Drinks-Only Shop - Cashier-Only Takeaway

Actor cycleCustomer orders drink at counter -> cashier creates takeaway order -> no waiter -> no kitchen/KDS required -> cashier takes cash -> cashier generates paid receipt PDF -> receipt print job queued, claimed, and marked printed -> cashier hands drink to customer.
Expected real shop behaviorA drinks-only counter should not need waiter or kitchen. Cashier must ring, print, collect payment, and hand over product quickly.
What happenedThe order became paid and receipt printing completed. However, staff-created takeaway currently still requires a table_id. The virtual counter table remained occupied after the paid takeaway order.
ResultPass for money/receipt flow, but operational caveat.
Production impactFor takeaway-only or drinks-only shops, this should be fixed before rollout or handled with a dedicated non-table counter order endpoint. Otherwise the counter/table UI may show stale occupied state even though the order is paid.

Scenario 4: Takeaway Cafe - Counter Order With Barista/KDS

Actor cycleCashier creates takeaway order -> cashier sends to KDS/barista -> kitchen/barista marks ready -> order sent to cashier -> cashier records payment -> paid receipt generated -> receipt print job completed -> product handed over.
Expected real shop behaviorUseful for a coffee shop where cashier takes order, barista prepares, cashier hands over.
What happenedKDS route, payment, paid receipt, and print job all passed.
ResultCore flow passes, same staff-takeaway table caveat applies.
Production impactUse customer/takeaway order path where possible, or update staff takeaway to work without a physical table and to clear virtual counters.

Scenario 5: Small Restaurant - Dine-in Waiter, Kitchen, Cashier

Actor cycleCustomer sits -> waiter enters restaurant order -> order sent to kitchen -> kitchen marks ready -> waiter sends to cashier -> cashier takes payment -> paid receipt PDF -> receipt print job completed -> table opens for next guest.
Expected real shop behaviorNormal small restaurant service with separate waiter, kitchen, and cashier roles.
What happenedOrder status paid. Payment status paid. Receipt printed at API queue level. Dine-in table returned to open.
ResultPass.
Readiness noteNeeds field validation with real kitchen screen, real receipt printer, and cashier drawer process.

Scenario 6: Big Restaurant - Table Move and Split Tender

Actor cycleWaiter opens dine-in table -> waiter moves order to another table -> KDS receives order -> kitchen marks ready -> waiter sends to cashier -> cashier settles with split cash/card tender -> paid receipt generated -> print queue completed -> moved-to table opens after payment.
Expected real shop behaviorLarger restaurants commonly move guests, combine operational steps, and accept split/mixed tenders.
What happenedTable move preserved active order flow. Split tender completed as paid. Receipt and print job completed. Final table status was open.

6. Production Blockers and Fix Priority

Priority Issue Why it matters Recommended action
P0 Staff takeaway orders require table_id and can leave virtual counter tables occupied after payment. Drinks-only and takeaway-only shops should not manage fake tables. A stale occupied counter can confuse cashier workflow and reporting. Add a staff counter/takeaway order endpoint that accepts branch_id without table_id, or update staff takeaway payment completion to release virtual tables consistently.
P0 Receipt scope is easy to misuse. Default full receipt after a paid order can show 0.00 due. Cashiers usually expect a paid receipt showing purchased items and amount paid. Cashier UI should call scope=paid for final receipts. Consider changing endpoint defaults after paid order, or expose explicit "Bill / Paid receipt / Reprint" buttons.
P0 No physical printer/card terminal validation in this run. API job status is not the same as printed paper, opened drawer, or captured terminal payment. Run field tests with actual receipt printer, kitchen printer/display, cash drawer, and payment terminal under normal, paper-out, network-loss, and retry cases.
P0 No load/concurrency proof. Large cafes/restaurants fail differently under 50+ active tickets, multiple devices, print contention, and network jitter. Add load tests for concurrent order creation, KDS updates, payments, receipt generation, and print job claiming.
P1 Offline is only partially proven. Top systems document offline orders/payments/KDS behavior. This project has idempotency and sync metadata but not a fully validated offline mobile queue. Define offline capability limits, persist local mobile actions, replay with conflict policy, and test internet outage recovery.
P1 Advanced restaurant modules are incomplete. Reservations, waitlist, labor scheduling, payroll/tips, gift cards, marketing campaigns, kiosk, delivery marketplace, and deep loyalty are expected by top platforms. Prioritize by target market. For small cafes/takeaway, fix counter flow and hardware first. For big restaurants, add reservations, labor, and advanced checks.
P1 UI acceptance testing is missing from this report. Backend may pass while staff still struggle during service if screens are slow, unclear, or too many taps. Run role-by-role Flutter and dashboard acceptance tests on real devices with the exact cycle in this report.

8. Recommended Next Engineering Steps

  1. Fix counter/takeaway order model: allow staff cashier order creation without table_id, or explicitly support virtual counters and release them after payment for takeaway/drinks flows.
  2. Harden receipt UX: make final receipt use paid scope by default after payment, keep bill/remaining-due scope separate, and add clear reprint behavior.
  3. Run hardware field test: real receipt printer, kitchen printer/display, cash drawer, card terminal, failed printer, paper-out, disconnected device, duplicate claim, retry, and printed audit trail.
  4. Add load tests: concurrent 50, 100, and 250 ticket simulations with KDS updates, payments, receipts, and print jobs.
  5. Run role UI acceptance: waiter, cashier, kitchen, owner, and customer apps on real devices, using the seven scenario scripts above.
  6. Define offline policy: what can happen offline, what cannot, how risk is shown to staff, how conflicts are replayed, and how failed payments are reconciled.
  7. Prioritize modules by target segment: for small cafe/takeaway, finish counter flow and hardware; for big restaurants, add reservations, labor/tips, seat/coursing, and manager approvals.
